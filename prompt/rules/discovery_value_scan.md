Gemini Search 任務：為 `/value_scan` 補找可驗證的價值重估證據、反證與資料缺口。

## 一、核心目的

`/value_scan` 的搜尋整理不是產出買賣建議，而是替後續報告判斷候選股是否真的具備「舊標籤 → 新標籤」的重估基礎。

搜尋結果必須分清楚：

1. 支持重估：官方公告、月營收、財報、法說會、公司 IR、產品、客戶、供應鏈、法人報告或資金流能支撐新標籤。
2. 支持反證：營收未跟上、毛利率下滑、庫存、訂單遞延、客戶集中、展望下修、題材過熱或估值過高。
3. 只作情緒：新聞熱度、論壇討論、法人短評、股價波動、未驗證傳聞。
4. 資料不足：缺少官方來源、缺少正文、日期不可驗證、只有 snippet、沒有營收或客戶資料支撐。

## 二、來源分類要求

請將來源優先分類為以下用途，不得混用：

| 分類 | 可支撐內容 | 限制 |
|---|---|---|
| 官方硬資料 | MOPS、TWSE、TPEx、月營收、財報、年報、重大訊息、公司 IR、法說會簡報 | 可作強證據，但仍需注意日期與是否為最新版本。 |
| 公司說法 | 公司官網、新聞稿、法說會口徑、投資人關係資料 | 可作公司展望，但不能單獨證明市場已重估。 |
| 法人或媒體解讀 | 主流財經媒體、法人報告摘要、產業媒體 | 可作輔助證據；若缺官方資料，不得標示 strong。 |
| 產業佐證 | 產業報告、供應鏈新聞、同業資料、客戶需求趨勢 | 可支持題材合理性，但需連回公司產品或營收。 |
| 籌碼資金 | 外資、投信、自營商、TDCC、融資券、大戶持股 | 只能輔助確認市場關注，不得單獨支撐高重估分。 |
| 反證風險 | 負面新聞、展望下修、毛利下滑、庫存、訂單遞延、客戶集中 | 必須保留，不得因候選股分數高而省略。 |
| 情緒或雜訊 | 論壇、社群、轉貼、低品質聚合站、標題農場 | 只能標示 sentiment，不得作為事實或強證據。 |

## 三、`/value_scan` finding 判斷規則

1. 若 finding 主張「支持重估」，必須至少有一個 Level 1 或 Level 2 來源，或本地結構化資料支撐。
2. 若只有新聞熱度、股價上漲、論壇討論或法人短評，stance 最多為 neutral，evidence_level 最多為 weak，並標示「只作情緒」。
3. 若缺少月營收、財報、法說會、產品、客戶或供應鏈資料，必須寫入 missing_data。
4. 若找到反證，必須以 negative 或 mixed finding 保留，不得只挑利多來源。
5. 若來源只有 snippet，且涉及營收、財報、客戶、訂單、CAGR 或毛利率，evidence_level 不得高於 weak。
6. 若同一股票同時有支持重估與反證，stance 應為 mixed，不能只寫 positive。
7. 若資料不足以確認「舊標籤 → 新標籤」，請寫「資料不足，無法確認重估」。

## 四、輸出補充要求

在每個 finding 中，請盡量明確說明該 finding 的用途：

- `supports_rerating`
- `supports_counter_evidence`
- `sentiment_only`
- `insufficient`

若通用 JSON schema 沒有專用欄位，請寫入 finding 或 reliability_note。
