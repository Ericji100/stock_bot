# AI 輸出品質與穩定性規則

本規則適用於所有會產生 AI 結果的指令。目標是讓報告精準、可靠、有品質、具備適度市場想像，且輸出格式穩定、可解析、可回查。

## 一、判斷層級

每個重要結論都要能歸入下列其中一類：

1. `verified_fact`：已有官方來源、公司公告、財報、月營收、法說會、可信產業來源或本地結構化資料支持。
2. `reasoned_inference`：由多個線索推導而來，但仍需要後續資料驗證。
3. `market_hypothesis`：市場可能交易的故事、題材擴散、資金想像或早期爆發前兆。
4. `sentiment_signal`：新聞熱度、社群討論、短線情緒或資金追逐跡象。
5. `insufficient`：資料不足、來源薄弱、日期不明或只有 snippet。

不得把 `reasoned_inference`、`market_hypothesis` 或 `sentiment_signal` 寫成 `verified_fact`。

## 二、固定分析骨架

報告型 AI 指令在每個核心章節都應盡量回答：

1. 目前確認的事實是什麼。
2. 市場可能怎麼解讀。
3. 如果要爆發，需要什麼條件。
4. 目前缺少什麼證據。
5. 哪些情況會讓判斷失效。
6. 結論屬於強證據、早期想像、題材擴散、籌碼點火、補漲候選、蹭題材風險或純財報型。

若某項資料不存在，請明確寫「目前資料不足」，並指出缺少的資料類型，不要用空泛語句帶過。

## 三、輸出穩定性

1. 必須遵守指定章節、表格與 JSON schema，不得任意新增大量章節。
2. 若指令要求 JSON，只輸出可被 `json.loads()` 解析的 JSON，不要輸出 Markdown code fence。
3. 若指令要求 Markdown 報告，標題層級要穩定，避免同一章節重複出現。
4. 不要在正文洩漏內部欄位名；必要欄位只能放在附錄或技術資料。
5. 結論段要短而清楚，避免大量重複來源摘要。

## 四、效率與成本

1. 中間整理 prompt 只做資料整理，不做完整投研判斷。
2. 報告 prompt 不要重複貼長規則；優先引用共用 rules。
3. 若資料量很大，優先保留來源 ID、日期、公司代號、反證、資料缺口與能影響結論的原文。
4. 不得為了節省篇幅刪除反證或資料缺口。

## 五、結論安全邊界

1. 不得輸出保證獲利、必漲、穩賺、無風險、目標價必到。
2. 不得把研究優先度寫成交易指令。
3. 不得因市場想像強就直接給高評分。
4. 不得因資料不足就完全省略可能的早期市場線索。
5. 若正反證據矛盾，必須正反並列，並說明目前採信哪一方與原因。
