# 資料來源可信度規則

本規則用來判斷來源能支撐到什麼程度。AI 必須把來源可信度、日期、正文完整度與用途分開寫清楚，避免把低品質來源當成強證據。

## 一、來源分級

| 等級 | 類型 | 可支撐內容 | 限制 |
|---|---|---|---|
| Level 1 | 官方來源：MOPS、TWSE、TPEx、公司公告、財報、月營收、法說會簡報、公司 IR | 已驗證事實、營收、財報、重大訊息、產品或客戶的正式揭露 | 仍需注意日期與是否為公司正式說法 |
| Level 2 | 主流財經媒體、產業媒體、法人報告、可信研究機構 | 產業趨勢、公司動態、法人觀點、供應鏈線索 | 不可直接當成公司已確認事實；法人目標價不得視為保證 |
| Level 3 | 一般新聞、部落格、整理型文章、二手轉述 | 背景脈絡、題材熱度、待驗證線索 | 需交叉驗證，不得單獨支撐高分 |
| Level 4 | 社群、論壇、YouTube、Threads、PTT、Dcard、討論區 | 市場情緒、散戶熱度、傳聞監控 | 只能作情緒或假說來源，不得當成事實 |
| Level 5 | 報價頁、排行榜、工具頁、無日期 snippet、內容農場、疑似 SEO 頁 | 通常應排除或只作檢索線索 | 不得支撐結論 |

## 二、正文完整度

每個來源需盡量判斷：

1. `full_text`：有完整正文，可引用摘要。
2. `partial_text`：只有部分正文，可作輔助。
3. `snippet_only`：只有搜尋摘要，不得作強證據。
4. `structured_data`：來自本地結構化資料，可依資料來源判斷可信度。
5. `failed` 或 `unknown`：不得支撐主要結論。

## 三、日期規則

1. 必須優先使用有 `published_date` 或可辨識日期的來源。
2. 若是歷史日期報告，來源日期不得晚於報告基準日，除非明確標示為回溯資料。
3. 若來源日期不明，只能作背景或待驗證線索。
4. 新聞入庫日期不等於新聞發布日期。

## 四、證據用途

每個 finding 應盡量標示用途：

1. `supports_rerating`：支持價值重估、新標籤、產品/客戶/營收改善。
2. `supports_counter_evidence`：支持降分、反證、風險、題材退燒。
3. `sentiment_only`：只代表市場情緒、新聞熱度或社群討論。
4. `insufficient`：資料不足，無法支撐判斷。

## 五、強證據條件

可以稱為強證據，通常需至少符合其中一項：

1. Level 1 官方來源直接支持。
2. Level 2 可信來源與本地結構化資料互相支持。
3. 多個獨立來源交叉驗證，且日期、主體、事件一致。
4. 來源能明確連到公司產品、客戶、營收、訂單、產能、毛利或法說會說法。

只有 snippet、單一新聞標題、社群討論、排行榜、股價波動或論壇傳聞，不得標示為強證據。

## 六、報告寫法

1. 重要結論需說明「來源層級 + 支撐內容 + 限制」。
2. 若缺少官方來源，請明確寫「缺少官方驗證」。
3. 若只有新聞或社群熱度，請寫「只能作為市場情緒或待驗證線索」。
4. 若來源互相矛盾，請列出矛盾點並降低結論確定性。
5. 不得引用無法回查的來源、憑空新增來源 ID、或把模型記憶當成來源。
